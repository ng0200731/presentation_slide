Design Proposal

Two approaches for implementing the multi-image gallery

Approach A: Self-Contained Gallery Element (Recommended)

One new element type: image-gallery

Data model:

  • content: JSON array of {url, caption} objects
  • styles.layout: "grid" | "linear"
  • styles.thumbSize: "small" | "medium" | "large"

Components:

  • ImageGalleryElement — renders grid/linear + empty state with drop zone
  • ImageGalleryToolbar — grid/linear toggle + size presets + add button

Upload flow:

  • New /api/upload endpoint accepts multipart, saves to server/uploads/, returns URL
  • Drop zone, paste handler, browse button all call same upload flow
  • URL appended to element content array

Pros: Clean separation, reusable, easy to extend

Cons: New element type needed in store + canvas

Approach B: Extend Existing ImageElement

Upgrade current image element

Data model changes:

  • content: change from single URL string to array of {url, caption}
  • Add layout/size style properties

Changes needed:

  • Refactor ImageElement to handle array content
  • Add toolbar controls to existing toolbar area
  • Backward-compatible: single URL auto-wraps to array

Pros: No new element type, backward compatible

Cons: Mixes single-image and gallery concerns in one component

Why I recommend A

  • Clean separation of concerns — gallery logic is complex (multi-upload, reordering, size presets, grid layout)
  • The existing ImageElement stays simple for single-image use cases
  • Easier to test and maintain independently
